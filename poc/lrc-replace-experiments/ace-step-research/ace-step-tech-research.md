## 二、核心架构

### 2.1 v1.0 架构（Diffusion + Linear Transformer）

```
用户输入（Tags + Lyrics + Duration）
         │
         ▼
   ┌─────────────────────────┐
   │  文本编码器              │
   │  - MERT（音乐理解）      │
   │  - m-HuBERT（多语言）    │
   │  REPA 对齐机制           │
   └────────────┬────────────┘
                │ 语义对齐特征
                ▼
   ┌─────────────────────────┐
   │  生成主干                │
   │  DCAE（深度压缩 AE）     │
   │  + 轻量化线性 Transformer│
   │  Flow Matching 生成      │
   └────────────┬────────────┘
                │
                ▼
          生成音频 latent
                │
                ▼ DCAE 解码
          输出音频（WAV/MP3）
```

**关键技术点**：
- **DCAE（Deep Compression AutoEncoder）**：来自 Sana，将音频压缩为极致高效的 latent 表示，是速度优势的核心
- **REPA（Representation Alignment）**：在训练时对齐 MERT/m-HuBERT 的语义表示，使模型「理解」音乐语义，实现歌词准确对齐
- **Flow Matching**：确定性 ODE 路径，生成比扩散模型更快、更稳定

### 2.2 v1.5 架构（LM + DiT 混合）

v1.5 在 v1.0 基础上引入了革命性升级：

```
用户输入（自然语言 query）
         │
         ▼
   ┌─────────────────────────┐
   │  Language Model（LM）   │
   │  - 「全能规划师」角色    │
   │  - Chain-of-Thought      │
   │  - 生成：歌词 + metadata │
   │    + captions（完整蓝图）│
   │  - 内在强化学习对齐      │  ← 无需外部奖励模型
   └────────────┬────────────┘
                │ Song Blueprint（歌曲蓝图）
                ▼
   ┌─────────────────────────┐
   │  Diffusion Transformer   │
   │  (DiT)                   │
   │  - 按蓝图合成音频        │
   │  - 支持 50+ 语言         │
   └────────────┬────────────┘
                │
                ▼
          完整歌曲音频
```

**v1.5 核心改进**：
- LM 作为规划器，实现从简单 query 到完整歌曲蓝图的自动生成
- 内在强化学习（intrinsic RL），无需人类偏好标注
- 速度：A100 上 4 分钟歌曲 < 2 秒（比替代方案快 10-120 倍）
- 内存：运行仅需 **< 4GB VRAM**（v1.0 需要 8GB）
- 支持 50+ 语言

## 三、与本项目强相关的功能

### 3.1 Lyric Editing（歌词编辑）⭐⭐⭐⭐⭐

**这是与「功能五：歌词替换与合成」最直接相关的能力。**

| 属性 | 详情 |
|------|------|
| **核心技术** | flow-edit（基于 Flow Matching 的局部编辑） |
| **能力描述** | 对已有歌曲中的局部歌词进行修改，同时**保持旋律、人声音色、伴奏不变** |
| **支持输入** | 生成内容 + **已上传的真实歌曲音频** |
| **中文支持** | ✅ 中文在支持语言中排名靠前 |
| **当前限制** | 每次只能修改小段歌词（否则音频产生失真），但可以多次叠加编辑 |
| **最新状态** | v1.0 已实现，v1.5 进一步提升质量 |

**flow-edit 工作原理**：

```
原曲音频 + 原歌词 + 新歌词
         │
         ▼ 编码为 latent
   原始 latent Z_original
         │
         ▼ 仅在目标时间段内：
   添加少量噪声 → Z_noisy
         │
         ▼ 用新歌词条件重建
   Z_reconstructed（新歌词，旋律保留）
         │
         ▼ 其他时间段保持 Z_original 不变
         │
         ▼ DCAE 解码
   输出音频（歌词已替换，旋律保持）
```

**核心优势**：
- **无跨域问题**：不需要「说话音频 → 歌唱音频」的转换，直接在音频 latent 空间操作
- **内置时间对齐**：旋律时间结构在 latent 中已被编码，替换歌词时自动遵循时间轴
- **端到端**：输入「原曲 + 新歌词」，直接输出修改后音频，无中间步骤

### 3.2 其他相关功能

| 功能 | 描述 | 与本项目的关联 |
|------|------|---------------|
| **Text2Music** | 从 Tags + Lyrics 生成完整歌曲 | 功能四（歌词生成）后续可集成 |
| **Lyric2Vocal（LoRA）** | 从歌词直接生成纯人声演唱 | 可作为 TTS→SVC 路线的升级替代 |
| **Repainting** | 局部重新生成歌曲特定时间段 | 配合 LRC 时间轴可实现逐句替换 |
| **Cover Generation**（v1.5）| 翻唱生成，保留原曲旋律 | 与 SongEcho 互补的另一种路径 |
| **Vocal-to-BGM**（v1.5）| 从人声生成配乐 | 与声音替换能力互补 |

## 五、安装与使用

### 5.1 安装方式

```bash
# 方式一：完整 clone（推荐开发调试）
git clone https://github.com/ace-step/ACE-Step.git
cd ACE-Step
conda create -n ace_step python=3.10 -y
conda activate ace_step
pip install -e .

# 方式二：pip 直接安装（API 集成用）
pip install git+https://github.com/ace-step/ACE-Step.git

# v1.5 版本
git clone https://github.com/ace-step/ACE-Step-1.5.git
```

### 5.2 macOS 特别注意

```bash
# macOS 必须关闭 bf16（否则报错）
acestep --bf16 false

# 推荐启动参数（省内存）
acestep --bf16 false --cpu_offload true --overlapped_decode true
```

### 5.3 API 调用示例（Lyric Editing）

```python
# ACE-Step v1.0 Lyric Editing 示意
from acestep import pipeline

result = pipeline.edit(
    audio_path="original_song.mp3",
    original_lyrics="[verse]\n不知道从哪里来\n也不知道要去哪里",
    new_lyrics="[verse]\n春风轻抚着脸庞\n带来远方的芬芳",
    edit_mode="only_lyrics",  # 只改歌词，保持旋律
    edit_strength=0.7,         # 编辑强度（0=完全保留原曲，1=完全重建）
    start_time=10.0,           # 编辑起始时间（秒）
    end_time=20.0,             # 编辑结束时间（秒）
)
result.save("output.mp3")
```

## 六、关键技术风险与已知限制

### 6.1 本项目场景的核心风险

| 编号 | 风险 | 严重度 | 可能性 | 详细分析 |
|------|------|--------|--------|---------|
| **R-1** | **对「已有真实歌曲」的 Lyric Edit 效果不达标** | 🔴 高 | 🟡 中 | ACE-Step 主要在 AI 生成内容上训练和优化，对真实录音（含环境音、人声特性）的编辑效果未经充分验证 |
| **R-2** | **中文歌词的声调保持问题** | 🔴 高 | 🟡 中 | 中文有 4 声，旋律音高与声调语调存在冲突。替换歌词时，新歌词的声调是否与原 F0 旋律兼容，无法保证 |
| **R-3** | **字数差异过大导致节奏错乱** | 🟡 中 | 🟠 高 | 新旧歌词字数差异 > 30% 时，模型可能压缩/拉伸音节到不自然程度 |
| **R-4** | **原音色是否能保持** | 🟡 中 | 🟡 中 | flow-edit 设计上保持旋律和伴奏，但人声音色是否精确保留，取决于 edit_strength 参数调优 |
| **R-5** | **逐句编辑的「接缝」问题** | 🟡 中 | 🟠 高 | 对每句歌词逐一 edit 后拼接，相邻句的衔接处可能存在音质突变 |
| **R-6** | **M2 Pro 上推理速度** | 🟢 低 | 🟢 低 | v1.5 只需 < 4GB 内存，速度实测应在 RTF 1.5~2.5× 范围内 |

### 6.2 已知官方文档限制

> 引自 ACE-Step 官方 README：
> - Lyric Editing 目前「只能修改小段歌词（否则会失真），但可以多次编辑叠加」
> - v1.5 支持 10 分钟以内的音乐生成（与需求文档「最长 10 分钟」对齐）

## 九、调研结论

### 9.1 总结

ACE-Step v1.5 是目前**最值得优先验证**的「歌词替换与合成」技术路线，原因如下：

1. **绕过了原方案的根本性难题**：不需要跨域（说话→歌唱）转换，直接在原曲 latent 空间操作
2. **端到端，流程极简**：输入「原曲 + LRC + 新歌词」，输出修改后音频，中间步骤最少
3. **质量已达商业水准**：v1.5 的 SongEval 自然度（4.59）已超越所有已测商业模型
4. **硬件友好**：< 4GB VRAM，M2 Pro 32GB 完全够用
5. **开源可商用**：Apache 2.0，无版权风险

### 9.2 核心待验证问题

以下 3 个问题是「能否采用 ACE-Step」的关键决策点，必须通过 PoC-B 验证后才能确定：

| 问题 | 验证方法 |
|------|---------|
| **Q1**: 对「真实中文录音」（非 AI 生成）的 Lyric Edit 效果是否可接受？ | exp-11（冒烟测试）|
| **Q2**: 中文歌词字数相差 ±3 字以内时，旋律保持度如何？ | exp-12（字数差异对比）|
| **Q3**: 逐句编辑拼接后整体连贯性是否足够好？ | exp-13（全曲流程）|
