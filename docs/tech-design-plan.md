## 概述

需求阶段已完成（见需求文档 v0.7），本阶段目标是完成技术方案设计，为编码开发做好准备。

整体分为三个阶段，按依赖关系顺序推进：

```
阶段一：技术选型调研  →  阶段二：整体架构设计  →  阶段三：开发准备
```

> **关键路径**：T-A4（歌词重合成方案）是整个项目技术难度最高、不确定性最大的模块，建议优先调研，并尽早完成 T-C1 的 PoC 验证，以规避后期返工风险。

---

## 阶段一：技术选型调研

**目标**：逐一澄清需求文档中 T-01~T-08 的技术决策项，输出选型结论。  
**输出文档**：`docs/tech-design.md` → 第 2 章（各模块技术选型）

| 任务编号 | 任务名称 | 对应需求决策项 | 关键问题 | 状态 |
|---|---|---|---|---|
| T-A1 | 运行环境确认 | T-08 | 目标 OS（macOS / Linux / Windows）、Python 版本、GPU 最低规格 | ✅ **已完成** |
| T-A2 | 人声分离方案选型 | T-04 | Demucs vs Spleeter vs UVR，质量 / 速度 / 本地运行能力对比 | 待开始 |
| T-A3 | 声音转换方案选型 | T-05 | RVC vs so-vits-svc，训练成本 / 推理质量 / 模型大小对比 | 待开始 |
| T-A4 | 歌词重合成方案选型 | T-06 | TTS 方案（VITS / BERT-VITS2 等）+ 旋律强制对齐方案；是否需要显式时间轴文件 | 待开始 |
| T-A5 | AI 歌词生成服务选型 | T-07 | OpenAI / Claude API vs 本地 LLM，成本 / 质量 / 离线能力 | 待开始 |
| T-A6 | 声音模型管理规范 | T-01 / T-02 / T-03 | 存储路径、文件格式、训练耗时预期、导入导出支持 | 待开始 |

**目标**：基于选型结论，设计整体系统架构和各模块的详细设计。  
**输出文档**：`docs/tech-design.md` → 第 3~5 章

| 任务编号 | 任务名称 | 内容说明 | 状态 |
|---|---|---|---|
| T-B1 | CLI 架构设计 | 命令结构、参数规范、入口设计（Typer / Click 等框架选择） | 待开始 |

## 任务依赖关系

```
T-A1 ──────────────────────────────────────┐
T-A2 ──┐                                   │
T-A3 ──┤                                   ▼
T-A4 ──┼──► T-B1（CLI 架构）               T-B5（UX 设计）
T-A5 ──┤    T-B2（模块接口）               │
T-A6 ──┘    T-B3（数据流）    ─────────────┼──► T-C1（PoC）
            T-B4（多语言）                  │    T-C2（脚手架）
                                            └──► T-C3（迭代计划）
```

**优先级建议**：
1. 先做 T-A1（环境）+ T-A2（人声分离），依赖少、风险低，可快速收敛
2. 并行推进 T-A3（声音转换）+ T-A4（歌词合成），两者均为高风险项，尽早调研
3. T-A4 结论出来后立即启动 T-C1 PoC 验证
4. 选型全部完成后，集中完成阶段二架构设计
5. 架构设计完成后进入阶段三，转入编码
