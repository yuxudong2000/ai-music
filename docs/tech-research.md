四项核心能力与技术模块的对应关系如下：

```
能力一：声音学习         ──► [人声分离] + [声音特征学习/Fine-tune]
```

## 一、能力一：声音学习（Voice Learning）

### 1.1 技术原理

声音学习的本质是**基于少量参考音频对预训练的声音转换模型进行个性化微调（Fine-tune）**，将目标人声的音色特征「固化」进模型权重，使其能够在推理阶段将任意人声「转换」成目标音色。

核心概念：

- **音色（Timbre）**：人声的「辨识度」——同样的音高和歌词，不同人唱出来的声音不同，差异主要来自共鸣腔形状、声带特征等物理因素，在频谱上体现为**频谱包络（Spectral Envelope）**的差异。声音转换模型所学习的，正是如何将一个人的频谱包络映射到另一个人。

- **说话人嵌入（Speaker Embedding）**：深度学习中将音色编码为固定维度向量的技术。模型在推理时提取参考音频的 Speaker Embedding，作为「目标音色」的数值表示。Fine-tune 的目标是让模型对该说话人的嵌入表征更加精准。

- **Zero-shot vs Fine-tune**：
  - *Zero-shot（零样本）*：只需在推理时实时提供参考音频（1~30 秒），模型直接从中提取 Speaker Embedding 完成转换，无需训练。速度快，但对音色的还原精度有上限。
  - *Fine-tune（微调）*：用目标人声的音频对预训练模型进行少量迭代训练，将该说话人的特征「固化」进模型，推理时只需加载微调后的模型权重，无需再提供参考音频。精度更高，但需要训练时间和存储。

本能力采用 **Fine-tune 模式**：用户提供参考音频 → 系统训练 → 保存个性化模型文件，后续声音替换直接加载该文件使用。

### 1.2 技术实现路径

```
输入：参考音频 MP3（含人声+伴奏）

步骤①  人声分离（M-1）
        ↓ 纯人声音轨（WAV/FLAC）
步骤②  音频预处理（切片、降噪、格式标准化）
        ↓ 干净的训练音频片段
步骤③  声音模型 Fine-tune（M-2）
        ↓ 个性化声音模型文件
输出：保存到本地的声音模型（如 ~/.ai-music/voices/jay_chou/）
```

> **说明**：步骤①中的人声分离是可选的预处理步骤——如果参考音频本身已是纯人声（如清唱录音），可直接跳过。实际中用户提供的往往是歌曲，故需分离。

### 1.3 核心技术决策点

| 决策点 | 问题 | 影响 |
|---|---|---|
| **D-1** | 选用哪个人声分离方案？（M-1） | 影响训练数据的纯净度，进而影响模型质量 |
| **D-2** | 选用哪个声音转换/学习方案？（M-2） | 影响声音相似度、训练时长、模型文件大小 |
| **D-3** | 声音模型的存储格式和路径规范？ | 影响模型管理、`voice update`、跨机器迁移 |
| **D-4** | 参考音频是否需要预先切片？如何切？ | 影响训练质量，需确定最优片段时长 |

## 二、能力二：声音替换（Voice Replacement）

### 2.1 技术原理

声音替换的核心是**人声分离 → 音色转换 → 重新混合**三段式处理，属于 SVC（Singing Voice Conversion，歌唱声音转换）任务的典型应用。

- **人声分离（Source Separation）**：混音歌曲中，人声与伴奏在频域上高度重叠，无法直接用滤波器分离。现代方案使用深度学习（如 Demucs 的 Hybrid Transformer）从时频域联合预测各音轨的「掩码（Mask）」，再应用掩码重建各音轨。分离产物：纯人声音轨（vocals）+ 伴奏音轨（accompaniment）。

- **音色转换（Voice Conversion）**：将分离出的人声音轨「换脸」——保持原始的**旋律（F0 音高曲线）**、**节奏**和**歌词**不变，只将**声道特征（频谱包络）**替换为目标音色。seed-vc 采用扩散（Diffusion）过程实现这一映射：以原始音频的语义内容（通过 Whisper 编码的内容特征）和原始 F0 为条件，在目标说话人的「声音空间」中采样，生成目标音色的演唱音频。

- **音频重混合**：用转换后的人声音轨替换原人声，与保持不变的伴奏音轨按适当音量比例混合，输出完整歌曲。

### 2.2 技术实现路径

```
输入：歌曲 MP3 + 声音名称

步骤①  人声分离（M-1）
        ↓ 纯人声音轨 + 伴奏音轨
步骤②  加载声音模型（读取 M-2 训练产物）
步骤③  声音转换推理（M-2）
        ↓ 转换后的人声音轨（音色已替换，旋律/节奏保持）
步骤④  音频混合（转换人声 + 原伴奏）
        ↓
输出：替换声音后的完整歌曲 MP3
```

### 2.3 核心技术决策点

| 决策点 | 问题 | 影响 |
|---|---|---|
| **D-1** | 人声分离方案（同能力一 M-1） | 伴奏分离质量直接影响最终输出 |
| **D-2** | 声音转换方案推理效果（M-2） | 音色相似度、旋律保持度、无伴奏情况下的音质 |
| **D-5** | 音频混合时是否需要音量均衡处理？ | 人声音量与伴奏音量的平衡 |

## 三、能力三：歌词生成（Lyrics Generation）

### 3.1 技术原理

歌词生成是一个**受约束的创意文本生成（Constrained Creative Text Generation）**任务。与自由写诗不同，歌词必须严格匹配原曲的段落结构，否则后续的旋律对齐将失败。

核心原理：

- **结构约束的来源**：歌曲的旋律由若干「乐句（Phrase）」组成，每个乐句对应歌词中的一句话。乐句的数量和时长在旋律录音中已固定，因此新歌词的段落数、每段句数必须与原歌词完全一致，否则新歌词多出的句子将找不到对应旋律，或原有旋律段落会空缺。

- **LLM 的结构遵从性（Instruction Following）**：现代大语言模型在足够明确的指令下，能够按照给定的段落/句子数量生成歌词。关键在于 Prompt 工程：需要将结构约束（第一段 4 句、第二段 4 句…）和创作要求（主题、风格、押韵）一同以结构化方式传入模型，并通过输出格式要求（如 JSON 或带标签的文本）方便解析和校验。

- **校验与重试**：LLM 输出的歌词不保证每次都严格符合句数约束，需要在系统层做结构解析和校验，不符合时自动重试（最多 N 次）或给出用户提示。

### 3.2 技术实现路径

```
输入：原歌词 TXT + 创作提示词

步骤①  歌词结构解析（本地逻辑）
        ↓ 约束信息：段落数 N、每段句数 [s1,s2,...]、每句字数 [...]
步骤②  Prompt 构建（结构约束 + 用户提示词 → 格式化 Prompt）
步骤③  LLM 调用（M-3）
        ↓ 生成的歌词文本
步骤④  结构约束校验（段落数、句数是否匹配，不匹配则警告）
步骤⑤  输出格式化（加 # [第X段] 注释标签）
        ↓
输出：新歌词 TXT 文件
```

### 3.3 核心技术决策点

| 决策点 | 问题 | 影响 |
|---|---|---|
| **D-6** | 选用哪个 LLM 服务？（M-3） | 生成质量、成本、离线能力 |
| **D-7** | Prompt 工程如何设计？ | 直接影响生成歌词的结构符合率和创作质量 |
| **D-8** | 生成失败/不符合约束时的重试策略？ | 影响用户体验和稳定性 |

## 四、能力四：歌词替换与合成（Lyrics Replacement & Re-synthesis）

### 4.1 技术原理

这是四项能力中**技术难度最高**、**不确定性最大**的模块。它需要将「语音内容（歌词）」与「旋律骨架（音高 + 时间轴）」解耦，再按新内容重新合成。

核心原理：

- **旋律的时间结构**：原曲中每句歌词都占据特定的时间段（如第一句在 12.5s~16.0s），这个时间段里旋律的音高轮廓（F0 曲线）是固定的。替换歌词后，新歌词的每个音节必须被安排到相应时间段内演唱，才能与旋律保持对齐。

- **音高提取（F0 Extraction）**：从原始人声音轨中提取逐帧的基频（F0）曲线，即旋律的「骨架」。这是重合成时的旋律约束，确保新歌词跟着原来的调子唱。

- **时间轴对齐（Forced Alignment）**：通过 ASR（自动语音识别）+ 强制对齐算法，得到原歌词每个字/词对应的时间戳，作为安排新歌词演唱时长的参考基准。

- **歌唱合成路线 A（声音转换路线）**：
  1. 用普通 TTS 生成新歌词的说话音频（获得正确的「内容」）
  2. 提取原人声的 F0 曲线（获得正确的「旋律骨架」）
  3. 用 SVC 模型以「新歌词内容 + 原 F0」为条件，合成「按原调演唱新歌词」的人声
  4. 按原时间轴对各句进行拼接，与伴奏重新混合

  这一路线的核心假设是：SVC 模型能够在保持 F0 的前提下，替换声道内容（即让模型「跟着原来的调子念出新歌词」）。其技术可行性是本项目最重要的 PoC 验证目标。

- **歌唱合成路线 B（SVS 路线）**：
  1. 从原始人声音轨中提取乐谱信息（音符序列 + 每个音符的起止时间和音高）
  2. 将新歌词与乐谱中的音符逐一对应（字-音对齐）
  3. 用 DiffSinger 等 SVS（Singing Voice Synthesis）系统，以「乐谱 + 新歌词」为输入，直接合成歌唱音频

  路线 B 的优势是合成质量上限更高（专为歌唱设计，音准精确），缺点是依赖「从音频中提取乐谱」这一前置步骤，该步骤本身技术难度较高，准确率有不确定性。

### 4.2 技术实现路径

这是四项能力中**技术难度最高**、**不确定性最大**的模块。核心挑战在于：如何让 TTS 合成出来的新歌词严格对齐原曲旋律的时间轴。

```
输入：歌曲 MP3 + 新歌词 TXT

步骤①  人声分离（M-1）
        ↓ 纯人声音轨 + 伴奏音轨
步骤②  原始歌词时间轴获取（M-4a）
        ↓ 每句歌词的时间戳（如 [00:12.5 → 00:16.0] 第一句）
        （路径一：解析用户提供的 LRC 文件；路径二：WhisperX 自动提取）
步骤③  歌唱 TTS 合成（M-4b）
        ↓ 按新歌词生成的演唱音频（含旋律 F0 引导）
步骤④  时间轴对齐与拉伸（M-4c）
        ↓ 将合成音频对齐到原时间轴
步骤⑤  音频混合（对齐后的新人声 + 原伴奏）
        ↓
输出：替换歌词后的完整歌曲 MP3
```

### 4.3 核心技术决策点

| 决策点 | 问题 | 影响 |
|---|---|---|
| **D-9** | 如何提取原始歌词时间轴？（M-4a） | 这是整个合成流程的时间参考基准 |
| **D-10** | 选用哪个歌唱 TTS 方案？（M-4b） | 合成音质、中文支持、对旋律的控制能力 |
| **D-11** | 如何将合成音频对齐到原时间轴？（M-4c） | 字数/音节差异时的处理策略（拉伸/压缩） |
| **D-12** | 是否需要用户显式提供时间轴对齐文件？ | 若系统无法自动推断，需新增 `lyrics align` 命令 |

## 五、各技术模块候选方案对比

### M-1：人声分离（Vocal Separation）

**需求**：从混音歌曲中分离出高质量的纯人声音轨（及伴奏音轨），且需在 Apple Silicon 本地运行。

| 方案 | 分离质量 | Apple Silicon 支持 | 安装方式 | 许可证 | 活跃度 |
|---|---|---|---|---|---|
| **Demucs v4（htdemucs）** | ⭐⭐⭐⭐⭐ 业界最高 | ✅ 支持 `-d mps` | `pip install demucs` | MIT | 活跃（Meta 维护） |
| Spleeter（Deezer） | ⭐⭐⭐ | ⚠️ 部分支持 | `pip install spleeter` | MIT | 较低（2021年后少更新） |
| UVR / audio-separator | ⭐⭐⭐⭐ | ✅ 支持 CoreML + arm64 | `pip install audio-separator` | MIT | 活跃（集成多模型） |

**方案说明**：

- **Demucs v4（htdemucs）**：Meta（Facebook AI）开源，采用 Hybrid Transformer 架构，在多个公开 benchmark 上质量排名第一；支持通过 `-d mps` 参数在 Apple Silicon 上启用 GPU 加速；`pip install demucs` 即可安装，Python API 友好。
- **audio-separator**：UVR（Ultimate Vocal Remover）的 Python 封装库，支持 MDX-Net、Demucs、VR Arch 等多种模型；可通过 Docker 在 arm64 上运行，也支持 Apple Silicon CoreML 加速；适合需要灵活切换模型的场景。
- **Spleeter**：Deezer 早期作品，速度快但质量逊于 Demucs v4，维护活跃度低，不优先推荐。

**🎯 建议选型：Demucs v4（htdemucs 模型）**  
质量最优，Apple Silicon MPS 加速已验证，安装简单，MIT 协议无商业限制。

**✅ 方案决策：Demucs v4（htdemucs 模型）**

| 决策项 | 内容 |
|---|---|
| **选定方案** | Demucs v4，使用 `htdemucs` 模型 |
| **安装方式** | `pip install demucs` |
| **调用方式** | Python API / CLI，MPS 加速：`-d mps` |
| **适用场景** | 能力一（参考音频预处理）、能力二（目标歌曲人声分离）、能力四（歌词替换合成前处理） |
| **输出格式** | WAV（44.1kHz，32-bit float）；后续可按需转换为其他格式 |
| **决策理由** | 分离质量业界最优（Hybrid Transformer 架构）；Apple Silicon MPS 加速已有社区验证；`pip install` 即可安装，Python API 友好；MIT 协议无商业限制；Meta 持续维护，活跃度高 |
| **遗留问题** | 歌唱音频中人声与部分乐器（如合唱、和声）频率范围重叠时分离效果有限，需在 PoC 阶段实测评估 |

### M-2：声音转换（Voice Conversion / Singing Voice Conversion）

**需求**：
- **学习阶段**：通过少量参考音频（1~5 分钟）对预训练模型进行 fine-tune，保存个性化模型
- **推理阶段**：将目标歌曲的人声音色转换为已学习的目标音色，同时保持旋律/音高不变
- Apple Silicon 本地运行，无独立 GPU

| 方案 | 零样本效果 | Fine-tune 支持 | 歌唱转换 | Apple Silicon | 模型大小 | 活跃度 |
|---|---|---|---|---|---|---|
| **seed-vc** | ⭐⭐⭐⭐⭐ SOTA | ✅ 最少 1 条语音 | ✅ 专用 SVC 模型（44.1kHz） | ✅ 2025-03 官方支持 | 200M（SVC） | 活跃 |
| RVC v2 | ⭐⭐⭐⭐ | ✅ 需较多数据 | ✅ 广泛使用 | ⚠️ 需手动配置 | ~200M | 活跃 |
| so-vits-svc | ⭐⭐⭐ | ✅ | ✅ | ⚠️ | ~300M+ | 较低 |

**方案说明**：

- **seed-vc**（推荐）：2024 年底发布，基于 Diffusion Transformer 架构。核心优势：
  - **零样本**（Zero-shot）能力强：仅需 1~30 秒参考音频即可克隆声音，无需 fine-tune
  - **专用歌唱 SVC 模型**：`seed-uvit-whisper-base`（200M 参数，44.1kHz 采样率），专为歌唱转换设计，在与 RVCv2 的客观评测对比中，SECS（说话人相似度）和 CER（字错率）全面优于 RVCv2
  - **Fine-tune 极低门槛**：最少 1 条语音，100 步（T4 上 2 分钟）即可完成，大幅提升特定说话人相似度
  - **2025-03 正式新增 Apple Silicon 支持**，提供专用 `requirements-mac.txt`
  - Python 3.10 官方推荐，与运行环境完全匹配

- **RVC v2**：成熟稳定，社区生态丰富（大量预训练模型可下载），但 Fine-tune 需要更多数据（建议 10 分钟以上），Apple Silicon 需手动配置环境；在歌唱转换的客观指标上整体逊于 seed-vc。

- **so-vits-svc**：2023 年流行，现维护活跃度降低，且在 seed-vc 发布后逐渐被取代，不推荐新项目使用。

**评测数据对比（歌唱声音转换 M4Singer 数据集）**：

|---|---|---|
| SECS（↑，说话人相似度） | **0.7405** | 0.7220 |
| CER（↓，字错率） | **19.70** | 23.15 |
| DNSMOS（↑，音质评分） | 3.12 | **3.28** |

> 注：seed-vc 在音色相似度和清晰度上优于 RVCv2，RVCv2 在原始音质分略优。

**🎯 建议选型：seed-vc（SVC 模型：seed-uvit-whisper-base）**  
零样本能力强（可作为不 fine-tune 的快速体验模式），fine-tune 门槛极低（满足"1~5 分钟参考音频"的需求），Apple Silicon 官方支持，与 Python 3.10 完全匹配。

**✅ 方案决策：seed-vc（SVC 模型：seed-uvit-whisper-base）**

| 决策项 | 内容 |
|---|---|
| **选定方案** | seed-vc，SVC 专用模型 `seed-uvit-whisper-base`（200M 参数，44.1kHz） |
| **安装方式** | `git clone` + `pip install -r requirements-mac.txt`（Apple Silicon 专用依赖） |
| **使用模式** | **零样本（Zero-shot）**：推理时直接传入参考音频，无需提前训练；**Fine-tune**：对目标说话人做少量迭代（100~500 步），进一步提升音色相似度 |

### M-3：LLM 歌词生成

**需求**：根据原歌词结构约束（段落数、句数、字数）和用户提示词，生成符合约束的中文歌词。

| 方案 | 中文质量 | 结构遵从性 | 成本 | 离线支持 | 接入复杂度 |
|---|---|---|---|---|---|
| **OpenAI GPT-4o API** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 按量付费（约 ¥0.01/千字） | ❌ 需联网 | 低 |
| **Anthropic Claude API** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 按量付费（类似 GPT-4o） | ❌ 需联网 | 低 |
| 本地 LLM（Qwen2.5 / DeepSeek-V2） | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 免费（需本地算力） | ✅ | 中（需 ollama 等管理） |

**方案说明**：

- **云端 API（GPT-4o / Claude）**：生成质量最优，结构约束遵从能力强，接入简单（`openai` SDK），成本低（单次歌词生成约 0.01~0.05 元人民币）。主要限制是需联网，不支持完全离线运行。
- **本地 LLM（Qwen2.5-7B / DeepSeek-V2-Lite）**：Apple Silicon（M 系列）可通过 `ollama` 或 `llama.cpp` 本地运行 7B 级模型，Qwen2.5-7B 中文能力较强，支持完全离线。但模型需占用 4~8 GB 内存，在 M 系列上响应较慢（约 10~30 秒/次生成）。

**🎯 建议选型：以 OpenAI / Claude API 为主，通过抽象接口支持未来接入本地模型**

**✅ 方案决策：可插拔的 LLM Provider 架构，支持云端与本地模型按配置切换**

| 决策项 | 内容 |
|---|---|
| **方案架构** | 统一 `LLMProvider` 抽象接口，通过配置文件（如 `~/.ai-music/config.yml`）中的 `llm.provider` 参数切换实现 |
| **云端模型支持** | **OpenAI**（GPT-4o / GPT-4o-mini）、**DeepSeek**（DeepSeek-V2 / DeepSeek-Chat）、**Qwen**（通义千问 API） —— 均兼容 OpenAI SDK 协议，通过 `base_url` + `api_key` + `model` 三参数即可适配 |
| **本地模型支持** | **Qwen2.5-7B**（通过 Ollama 运行）等，通过 Ollama 的 OpenAI 兼容 API（`http://localhost:11434/v1`）接入，无需额外适配 |
| **默认配置** | 首次使用时引导用户配置 API Key，默认推荐 DeepSeek（性价比高，中文能力强，国内访问无障碍） |
| **配置文件示例** | 见下方 |
| **决策理由** | OpenAI、DeepSeek、Qwen 的 API 均兼容 OpenAI SDK 协议，只需一套 `openai` Python SDK + 不同 `base_url` 即可覆盖所有云端服务；Ollama 本地也暴露同协议 API，本地/云端切换零代码改动；用户按自身网络环境和成本偏好自选 |
| **遗留问题** | Prompt 工程需针对不同模型微调（各模型对结构约束的遵从能力有差异）；本地 7B 模型在歌词创作质量上预计不如云端大模型 |

```yaml
  # base_url: https://api.openai.com/v1    # OpenAI
```

### M-4：歌唱重合成（Singing Re-synthesis）

这是**技术风险最高**的模块，目前的开源生态尚在快速演进中。

#### M-4a：歌词时间轴获取

**目标**：获取每句歌词对应的时间戳（开始/结束时间），作为重合成的时间参考基准。

> **关键结论**：时间轴的获取分两条路径，优先使用用户提供的 LRC 文件；仅当无 LRC 文件时，才需要通过 ASR + 强制对齐从音频中自动提取。

**路径一：用户提供 LRC 文件（推荐，优先）**

如果用户已有歌曲对应的 LRC 歌词文件（大多数流行歌曲可从网易云、QQ 音乐等平台下载获得），系统直接解析 LRC 时间戳，**完全跳过 ASR 步骤**。

- 优点：精度高（人工校对过）、速度快（无需模型推理）、无额外依赖
- LRC 格式示例：
  ```
  [00:12.50]第一句歌词
  [00:16.00]第二句歌词
  [00:21.30]第三句歌词
  ```

**路径二：无 LRC 文件时，通过 ASR + 强制对齐自动提取（兜底）**

当用户无法提供 LRC 文件时，系统从人声音轨中自动识别歌词并推断时间戳。

| 方案 | 方法 | 中文支持 | 精度 | Apple Silicon |
|---|---|---|---|---|
| **WhisperX** | Whisper ASR + phoneme-level forced alignment | ✅ | 词级别时间戳 | ✅ |
| Montreal Forced Aligner（MFA） | 传统声学模型强制对齐 | ✅（需中文模型） | 音素级别 | ✅ |
| Whisper（原版） | ASR 输出 word timestamp | ✅ | 词级别（精度较低） | ✅ |

路径二选用 **WhisperX**：在 Whisper 基础上增加了强制对齐步骤，时间戳精度更高，支持中文，可本地运行。

**✅ 方案决策：双路径策略**

| 决策项 | 内容 |
|---|---|
| **首选路径** | 用户提供 LRC 文件 → 直接解析，跳过 ASR |
| **兜底路径** | 无 LRC 文件 → WhisperX 自动提取（`pip install whisperx`） |
| **CLI 体现** | `lyrics-replace` 命令新增可选参数 `--lrc`；若未提供，系统自动调用 WhisperX |
| **决策理由** | LRC 文件获取成本极低（主流平台均可下载），且精度远优于自动提取；两条路径互为补充，对用户透明 |
| **遗留风险** | WhisperX 在歌唱场景（音高波动大）识别准确率低于说话场景，自动提取精度需 PoC 实测 |

#### M-4b：歌唱 TTS 合成

**目标**：给定歌词文本和旋律 F0 曲线（从原人声提取），合成出与原旋律匹配的新人声演唱。

| 方案 | 中文歌唱 | 旋律控制 | 质量 | 本地运行 | 成熟度 |
|---|---|---|---|---|---|
| **seed-vc SVC**（M-2 复用） | ✅ | ✅（F0 条件化） | ⭐⭐⭐⭐ | ✅ | 活跃 |
| DiffSinger | ✅ | ✅ 需乐谱（SVS） | ⭐⭐⭐⭐⭐ | ✅ | 活跃 |
| So-VITS-SVC | ✅ | ✅ | ⭐⭐⭐ | ✅ | 较低 |
| ACE-Step（2025） | ✅ | ✅ | ⭐⭐⭐⭐⭐ | ✅ | 最新 |

**核心挑战与两种技术路线**：

> **路线 A（声音转换路线，推荐）**：  
> 不从零生成演唱，而是先用 TTS 生成新歌词的普通说话音频，再用 seed-vc 的 SVC 模型将其"转换"成歌唱风格，同时以原人声的 F0 为旋律引导。这样可以复用能力一/二的 seed-vc 模型，架构一致性好。
>
> **路线 B（SVS 路线）**：  
> 直接用专门的歌唱语音合成（Singing Voice Synthesis）系统（如 DiffSinger）生成歌唱音频，需要乐谱（音符+时长）作为输入，精度更高但依赖精确的旋律分析。

**🎯 建议**：优先采用**路线 A（声音转换路线）**，复用 seed-vc SVC 模型。PoC 验证通过后，若质量不满足要求，再评估路线 B（DiffSinger）。

**✅ 方案决策：同时支持路线 A 和路线 B，通过配置参数选择**

| 决策项 | 内容 |
|---|---|
| **路线 A（默认）** | seed-vc SVC 路线：TTS 生成说话音频（使用 **edge-tts** 或 **pyttsx3**，见下方说明）→ 提取原人声 F0 → seed-vc SVC 以新歌词内容 + 原 F0 为条件合成演唱 |
| **路线 B（备选）** | DiffSinger SVS 路线：从原人声提取乐谱（音符 + 时长）→ DiffSinger 直接生成歌唱音频 |
| **切换方式** | `~/.ai-music/config.yml` 中的 `synthesis.route` 参数，可选值：`svc`（路线 A）/ `svs`（路线 B） |
| **默认路线** | `svc`（路线 A），复用 M-2 的 seed-vc SVC 模型，架构一致、依赖少 |
| **路线 A 适用场景** | 歌词字数与原曲接近、对合成质量要求中等时，推荐使用 |
| **路线 B 适用场景** | 路线 A 质量不满足要求、或原曲旋律结构较复杂（如美声、民谣等对音准要求高的曲风）时切换 |
| **决策理由** | 两条路线技术成熟度和适用场景不同，互为补充；通过配置切换对用户透明，无需修改命令；路线 A 可复用已有 seed-vc 模型权重，减少额外依赖；路线 B 作为质量兜底保留扩展空间 |
| **遗留风险** | 路线 A 的核心可行性（SVC 模型能否在 F0 引导下正确演唱新歌词）是最高优先级 PoC 验证目标；路线 B 依赖 DiffSinger 的乐谱提取精度，需额外实测 |

**配置文件示例**（`~/.ai-music/config.yml`）：

```yaml
synthesis:
  # 歌唱重合成路线
  # svc: 路线 A，seed-vc SVC 声音转换路线（默认）
  # svs: 路线 B，DiffSinger 歌唱语音合成路线
  route: svc
```

**接口设计**：将两条路线封装为可替换的 `SynthesisBackend` 抽象：

> **路线 A 中 TTS 引擎的选择**：路线 A 的第一步需要将新歌词合成为普通说话音频（作为 SVC 的输入内容）。TTS 引擎的选择不影响最终音色（音色由 SVC 模型决定），但影响发音准确率和韵律自然度。
>
> | TTS 引擎 | 中文支持 | 离线 | 安装 | 说明 |
> |---|---|---|---|---|
> | **edge-tts** | ⭐⭐⭐⭐⭐ | ❌ 需联网 | `pip install edge-tts` | 微软 Azure TTS 免费接口，中文发音质量高 |
> | **pyttsx3** | ⭐⭐⭐ | ✅ | `pip install pyttsx3` | 调用系统 TTS（macOS 用 AVSpeechSynthesizer），无需联网 |
> | **kokoro-tts** | ⭐⭐⭐⭐ | ✅ | `pip install kokoro` | 2025 年新兴开源 TTS，本地运行，质量接近云端 |
>
> **建议**：默认使用 **edge-tts**（质量最优，安装最简单）；用户无网络时自动 fallback 到 **pyttsx3**（macOS 系统内置）。

```python

    def synthesize(self, lyrics: str, f0: np.ndarray, alignment: LyricsAlignment) -> np.ndarray: ...

class SVCBackend(SynthesisBackend):
    """路线 A：seed-vc SVC 路线"""
    ...

class SVSBackend(SynthesisBackend):
    """路线 B：DiffSinger SVS 路线"""
    ...
```

#### M-4c：时间轴对齐与音频拉伸（Time Alignment & Time-stretching）

**目标**：将 M-4b 合成出的新人声音频，按照 M-4a 提供的原始时间轴，逐句拉伸或压缩，使每句新歌词的演唱时长与原旋律时间段精确对齐。

**核心问题**：即使新旧歌词字数完全一致，TTS 合成的每句音频时长也不会精确等于原时间段的时长——因为 TTS 按**说话韵律**合成，每个字的发音时值不固定，且不会自动匹配原曲的音乐节拍。例如原曲某句占 3.5 秒，合成出的该句音频可能是 3.1 秒或 3.8 秒，仍需拉伸/压缩对齐且不改变音高。

> **与字数约束的关系**：如果在歌词生成阶段已保证新旧歌词**每句字数基本一致**，则 M-4c 的拉伸比例会更接近 1.0x（例如从可能的 0.65x 收窄到 0.85x~1.15x 范围），音质损失更小，超限风险接近消除。因此，歌词字数约束和 M-4c 是互补关系：字数约束降低了对齐难度，M-4c 做最终的精确对齐。

| 方案 | 方法 | 音质 | Apple Silicon | 说明 |
|---|---|---|---|---|
| **librosa PSOLA / time_stretch** | 相位声码器时间伸缩 | ⭐⭐⭐⭐ | ✅ | `librosa.effects.time_stretch()`，纯 CPU，无需 GPU |
| **pydub + ffmpeg atempo** | FFmpeg atempo 滤镜 | ⭐⭐⭐ | ✅ | 速率调整有上限（0.5x~2.0x），超出需多次叠加 |
| **Rubber Band Library（pyrubberband）** | 高质量时间伸缩 | ⭐⭐⭐⭐⭐ | ✅ | `pip install pyrubberband`，专业音频级别质量 |

**方案说明**：

- **pyrubberband**：Rubber Band Library 的 Python 封装，专为音乐/人声设计，在时间拉伸时能保持音高和音色，处理歌唱音频质量明显优于通用算法，是广播、DAW 软件中的主流方案。
- **librosa time_stretch**：基于相位声码器，适合小范围拉伸（±20% 以内），超出范围会出现明显金属味。
- **ffmpeg atempo**：速度简单可靠，但音质不如 Rubber Band，更适合说话音频。

**✅ 方案决策：pyrubberband（Rubber Band Library）**

| 决策项 | 内容 |
|---|---|
| **选定方案** | pyrubberband（`pip install pyrubberband`） |
| **处理策略** | 按 M-4a 的每句时间戳，逐句独立拉伸/压缩合成音频，使其时长与原时间段匹配 |
| **允许偏差范围** | 拉伸比例建议控制在 **0.7x ~ 1.4x** 以内，超出范围说明新旧歌词字数差异过大，需警告用户 |
| **音高保持** | Rubber Band 时间伸缩过程保持音高不变（pitch-preserving），不影响旋律 |
| **决策理由** | 专业音频级别的时间伸缩质量，处理歌唱音频时音色保真度最优；Python 接口简单；Apple Silicon 纯 CPU 可运行，无需 GPU |
| **遗留风险** | TTS 合成时长与原时间段仍有偏差（即使字数一致，约 ±10%~20%），但只要歌词生成阶段保证了字数约束，拉伸比例通常可控制在 0.8x~1.2x 的安全范围内，音质损失可忽略不计 |

| R-1 | 歌唱重合成（M-4b）质量不达标，新歌词音调/节奏与原曲明显不匹配 | 高 | 中 | 尽早 PoC，备选路线 B（DiffSinger） |
| R-2 | Apple Silicon MPS 对某些算子不支持，导致推理报错 | 中 | 低 | 回退 CPU 运行，但速度降低 |
| R-3 | seed-vc fine-tune 在 Apple Silicon 无 GPU 情况下训练时间过长（> 30 分钟/1分钟数据） | 中 | 中 | 降低 max-steps，优先推荐零样本模式 |
| R-4 | WhisperX 在歌唱场景（非说话场景）的时间戳精度不足 | 中 | 中 | 对歌唱音频做预处理（音高平滑），或手动校正 |
| R-5 | LLM API 网络不可用（国内网络限制） | 低 | 中 | 支持 Base URL 配置，兼容国内代理/中转 |

## 七、技术选型汇总

| 模块 | 选型结论 | 备注 |
|---|---|---|
| **运行环境** | macOS Apple Silicon + Python 3.10 + PyTorch MPS | 已确定 |
| **M-1 人声分离** | **Demucs v4（htdemucs 模型）** | `pip install demucs`，`-d mps` 加速 |
| **M-2 声音转换** | **seed-vc**（SVC 模型：seed-uvit-whisper-base） | 零样本 + fine-tune，官方支持 Apple Silicon |
| **M-3 LLM 歌词生成** | **DeepSeek / OpenAI / Qwen**（云端）/ **Ollama**（本地） | 统一 OpenAI SDK 协议，通过 `llm.provider` 配置切换 |
| **M-4a 时间轴获取** | **LRC 文件解析**（首选）/ **WhisperX**（无 LRC 时兜底） | 优先用用户提供的 LRC，无 LRC 时自动 ASR |
| **M-4b TTS 引擎** | **edge-tts**（默认）/ pyttsx3（离线 fallback） | 路线 A 内部步骤；音色由后续 SVC 决定 |
| **M-4b 歌唱合成** | **seed-vc SVC 路线 A**（默认）/ DiffSinger 路线 B（备选） | 通过 `synthesis.route` 配置切换 |
| **M-4c 时间轴对齐** | **pyrubberband** | 音高保持时间伸缩，`pip install pyrubberband` |
| **CLI 框架** | **Typer**（基于 Click） | 类型注解友好，帮助文档自动生成 |
| **音频处理基础库** | **pydub + ffmpeg** | 格式转换、音频混合 |
