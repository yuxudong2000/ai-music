# AI Music 工具 - 需求描述文档

> **版本**：v0.8  
> **创建日期**：2026-03-16  
> **状态**：草稿

---

## 1. 项目概述

本项目旨在开发一款命令行音乐处理工具，基于 AI 技术，提供五项核心能力：

1. **LRC 提取（LRC Extraction）**：从歌曲音频中自动提取歌词并生成带时间戳的 LRC 歌词文件，也支持直接导入已有 LRC 文件进行格式校验与规范化。
2. **声音学习（Voice Learning）**：通过提供一个或多个参考音频，让系统学习并保存目标人声的音色模型，供后续反复使用。
3. **声音替换（Voice Replacement）**：在不改变歌曲伴奏、音效及其他音频属性的前提下，将歌曲中的人声替换为已学习的目标人声音色。
4. **歌词生成（Lyrics Generation）**：用户提供原歌词文本作为结构约束来源，由 AI 自动生成符合约束的新歌词文件，用户可在此基础上进一步手动编辑完善。
5. **歌词替换与合成（Lyrics Replacement & Re-synthesis）**：在保持原有旋律和人声音色不变的前提下，将歌曲中的歌词替换为指定歌词文件中的新歌词，并重新合成完整歌曲音频。**LRC 文件为必须输入**，需在执行本功能前通过功能一预先准备好。

**目标用户**：个人娱乐用户、独立音乐创作者  
**交付形式**：命令行工具（CLI）  
**第一期语言支持**：中文（设计需考虑未来扩展英文等多语言）

## 2. 功能需求

### 2.1 功能一：LRC 提取（LRC Extraction）

#### 2.1.1 功能描述

LRC 文件是歌词替换与合成的关键输入，记录了每句歌词对应的精确时间戳，是后续旋律对齐的基础。本功能提供两种获取 LRC 文件的途径：

- **路径一（推荐）：导入已有 LRC 文件**：大多数流行歌曲可从网易云、QQ 音乐等平台直接下载 LRC 文件，系统对其进行格式校验与规范化，生成标准 LRC 文件。
- **路径二（兜底）：从音频自动提取**：当用户无法获取 LRC 文件时，系统通过 ASR（自动语音识别）+ 强制对齐算法，从歌曲音频中自动提取歌词及时间戳，生成 LRC 文件。

> **为什么作为独立功能**：LRC 提取是歌词替换流程中耗时较长（自动提取模式）且结果需人工确认的步骤。将其独立出来，允许用户提前准备、检查和修正 LRC 文件，避免在歌词替换主流程中等待或反复重跑。同时，同一首歌的 LRC 文件可多次复用，无需重复提取。

#### 2.1.2 输入规格

| 项目 | 规格 |
|---|---|
| 输入音频格式（路径二） | MP3（后续可扩展） |
| 输入 LRC 文件（路径一） | 标准 LRC 格式文件（`.lrc`） |

#### 2.1.3 输出规格

| 项目 | 规格 |
|---|---|
| 输出文件格式 | LRC 文件（`.lrc`，UTF-8 编码） |
| 时间戳精度 | 句级别（每句歌词对应一个开始时间戳） |
| 输出内容 | `[MM:SS.xx]歌词文本` 格式，每行一句 |

#### 2.1.4 LRC 文件格式规范

系统生成和接受的 LRC 文件遵循以下格式：

```
[00:12.50]第一句歌词
[00:16.00]第二句歌词
[00:21.30]第三句歌词
[00:25.80]第四句歌词
```

- 每行格式：`[分:秒.厘秒]歌词文本`
- 时间戳精度至少到厘秒（0.01 秒）
- 纯音乐段落（间奏、前奏等）不强制要求标注，但允许用空行或 `[MM:SS.xx]` 标记占位

#### 2.1.5 CLI 接口示例（示意，非最终设计）

```bash
# 路径一：导入已有 LRC 文件（格式校验 + 规范化）
ai-music lrc import \
  --lrc downloaded.lrc \
  --output song.lrc

# 路径二：从音频自动提取 LRC（ASR + 强制对齐）
ai-music lrc extract \
  --input song.mp3 \
  --output song.lrc

# 查看 LRC 文件内容（带时间轴预览）
ai-music lrc preview --lrc song.lrc
```

### 2.2 功能二：声音学习（Voice Learning）

#### 2.2.1 功能描述

用户提供一个或多个包含目标人声的参考音频文件，系统对这些音频进行学习，提取并保存音色特征模型，并为其命名。保存后的声音模型可在功能三（声音替换）中反复使用，无需每次重新提供参考音频。

#### 2.2.2 声音模型来源

| 来源 | 描述 |
|---|---|
| **用户自定义学习** | 用户提供一个或多个目标人声的参考音频（如目标歌手的录音片段），系统学习并保存为命名的声音模型 |
| **系统预设** | 系统内置若干预设声音模型，随工具一起分发，无需用户手动学习注册 |

无论来源如何，所有声音模型在使用时均通过统一的**声音名称（Voice Name）**引用。

#### 2.2.3 输入规格

| 项目 | 规格 |
|---|---|
| 参考音频格式 | MP3（后续可扩展） |
| 参考音频数量 | 支持传入 1 个或多个文件（多个文件有助于提升模型质量） |
| 参考音频时长建议 | 每段建议不低于 10 秒，总时长建议 1~5 分钟 |
| 声音名称 | 用户自定义，仅允许字母、数字、下划线、中文，唯一标识 |

#### 2.2.4 CLI 接口示例（示意，非最终设计）

```bash
# 学习单个参考音频并保存为声音模型
ai-music voice learn \
  --name "jay_chou" \
  --ref ref_audio.mp3

# 学习多个参考音频（提升模型质量）
ai-music voice learn \
  --name "jay_chou" \
  --ref ref1.mp3 ref2.mp3 ref3.mp3

# 向已有声音模型追加新的参考音频（更新/增强模型）
ai-music voice update \
  --name "jay_chou" \
  --ref new_ref1.mp3 new_ref2.mp3

# 查看所有已保存的声音模型（含用户自定义与系统预设）
ai-music voice list

# 删除用户自定义的声音模型
ai-music voice remove --name "jay_chou"
```

### 2.3 功能三：声音替换（Voice Replacement）

#### 2.3.1 功能描述

用户输入一首歌曲音频，并指定一个已通过功能二学习保存的声音模型名称（或系统预设声音），工具自动将歌曲中的人声音色替换为目标声音，同时保持伴奏、混响、音效等所有其他音频元素完全不变。

#### 2.3.2 目标声音来源

| 来源 | 描述 |
|---|---|
| **用户自定义模型** | 通过功能二 `voice learn` 学习并保存的声音模型，通过名称引用 |
| **系统预设模型** | 系统内置的预设声音模型，通过名称引用，与自定义模型接口完全一致 |

#### 2.3.3 处理约束

- 伴奏、背景音乐、混响、音效等非人声部分**完全保留，不做任何修改**
- 替换后的人声需与原曲节奏、旋律时间轴保持对齐
- 第一期仅支持**中文歌曲**，系统架构需预留多语言（英文等）扩展接口

#### 2.3.4 输入输出规格

| 项目 | 规格 |
|---|---|
| 输入音频格式 | MP3（后续可扩展） |
| 输出音频格式 | MP3 |
| 音频时长限制 | 最长 **10 分钟** |

#### 2.3.5 CLI 接口示例（示意，非最终设计）

```bash
# 使用用户自定义学习的声音替换
ai-music voice-replace \
  --input song.mp3 \
  --voice "jay_chou" \
  --output output.mp3

# 使用系统预设声音替换
ai-music voice-replace \
  --input song.mp3 \
  --voice "preset_male_01" \
  --output output.mp3
```

### 2.4 功能四：歌词生成（Lyrics Generation）

#### 2.4.1 功能描述

用户**直接提供**歌词文本，并提供创作主题/风格等提示词，由 AI 依据歌词结构约束生成新歌词，并输出为可编辑的文本文件。用户可在此基础上对歌词进行手动修改和完善。

#### 2.4.2 歌词结构来源

系统支持以下方式提供歌词结构约束：

| 来源方式 | 描述 | 适用场景 |
|---|---|---|
| **直接输入歌词文本** | 用户提供原歌词文本文件（TXT 格式），系统从中提取段落数、句数、字数等结构约束 | 已有歌词文本、无需处理音频时 |


#### 2.4.3 歌词生成方式

| 方式 | 描述 |
|---|---|
| **AI 自动生成** | 用户提供主题、风格、关键词等提示词，由 AI 依据结构约束自动生成新歌词 |
| **结构模板导出** | 系统仅导出原歌词的结构模板（空歌词框架），供用户完全手动填写 |

#### 2.4.4 歌词结构约束

AI 生成歌词时，系统从歌词来源中提取以下约束并传递给 AI；若用户导出结构模板后手动填写的歌词不满足段落数量或句数约束，系统应在执行前给出警告，列出具体差异，并提示用户确认是否继续。

| 约束项 | 要求 | 说明 |
|---|---|---|
| **段落数量** | 必须与原歌词相同 | 每个段落对应一段旋律，数量变化会导致旋律错位 |
| **每段句数** | 必须与原歌词相同 | 每句对应一条旋律线，句数变化会导致旋律空白或截断 |
| **每句字数/音节数** | 应尽量与原歌词保持一致 | 字数差异过大会导致发音被拉伸或压缩，影响合成质量 |
| **押韵/韵脚** | 可选，不强制要求 | 不影响技术合成，但影响歌词的音乐性和听感 |
| **语义/主题** | 不限制 | 完全由用户自由指定 |

#### 2.4.5 输出歌词文件格式

生成的歌词文件为 **TXT 格式（UTF-8 编码）**，每行一句歌词，段落之间以空行分隔，并保留结构注释（如 `# [第一段]`），方便用户对照编辑：

```
# [第一段]
第一句歌词
第二句歌词
第三句歌词

# [第二段]
第四句歌词
第五句歌词
```

#### 2.4.6 CLI 接口示例（示意，非最终设计）

```bash
# 方式一：直接输入原歌词文本，AI 生成新歌词
ai-music lyrics generate \
  --lyrics original_lyrics.txt \
  --prompt "写一首关于夏天海边的歌词" \
  --output new_lyrics.txt

# 方式二：从歌词文本导出结构模板（空框架，供手动填写）
ai-music lyrics template \
  --lyrics original_lyrics.txt \
  --output lyrics_template.txt
```

### 2.5 功能五：歌词替换与合成（Lyrics Replacement & Re-synthesis）

#### 2.5.1 功能描述

用户输入一首歌曲音频、一个新歌词文件（可来自功能四生成后编辑的结果，也可完全手动创作）以及一个 **LRC 时间轴文件（必须输入，需通过功能一预先准备）**，工具自动将原歌词替换为新歌词，并基于 LRC 文件提供的旋律时间轴和原歌手音色，重新合成包含新歌词的完整歌曲音频。

> **为什么 LRC 必须提前准备**：LRC 文件的自动提取（ASR）在歌唱场景下精度有限，结果需要用户人工核对和修正。将 LRC 准备作为独立的前置步骤（功能一），既保证了时间轴精度可控，也避免在合成主流程中因 ASR 错误导致整个流程失败。

#### 2.5.2 歌词文件来源

| 来源 | 描述 |
|---|---|
| **功能四生成后编辑** | 使用 `lyrics generate` 或 `lyrics template` 命令生成文件，在文本编辑器中手动调整后使用 |
| **完全手动创作** | 用户完全自行编写歌词文件，按照规定格式保存后传入 |

#### 2.5.3 处理约束

- 原曲旋律、伴奏、原歌手音色**完全保留，不做任何修改**
- AI 合成新歌词时，需基于 LRC 文件中的时间轴进行旋律对齐
- 第一期仅支持**中文歌曲**，系统架构需预留多语言（英文等）扩展接口

#### 2.5.4 输入输出规格

| 项目 | 规格 |
|---|---|
| 输入音频格式 | MP3（后续可扩展） |
| 输入歌词格式 | TXT 文本文件（UTF-8 编码） |
| 输入 LRC 格式 | LRC 文件（`.lrc`，**必须提供**） |
| 输出音频格式 | MP3 |
| 音频时长限制 | 最长 **10 分钟** |

#### 2.5.5 CLI 接口示例（示意，非最终设计）

```bash
# LRC 为必须参数，需通过功能一（lrc import 或 lrc extract）提前准备
ai-music lyrics-replace \
  --input song.mp3 \
  --lyrics new_lyrics.txt \
  --lrc song.lrc \
  --output output.mp3
```

### 2.6 功能组合使用

五项核心能力作为**独立命令**提供，用户可灵活组合。典型工作流如下：

**工作流 A：只换声音**
```
① voice learn（声音学习，首次使用时执行一次）→ ② voice-replace（声音替换）
```

**工作流 B：只换歌词（有现成 LRC 文件）**
```
① lrc import（导入/校验 LRC 文件）
② lyrics generate（AI 生成歌词）→ 手动编辑歌词文件
③ lyrics-replace --lrc（歌词替换合成）
```

**工作流 C：只换歌词（需自行提取 LRC）**
```
① lrc extract（从音频提取 LRC，检查并修正结果）
② lyrics generate（AI 生成歌词）→ 手动编辑歌词文件
③ lyrics-replace --lrc（歌词替换合成，指定步骤①提取的 LRC）
```

**工作流 D：换声音 + 换歌词**
```
① voice learn（声音学习）
② lrc import 或 lrc extract（准备 LRC 文件）
③ lyrics generate（生成歌词）→ 手动编辑
④ lyrics-replace --lrc（歌词替换合成）
⑤ voice-replace（声音替换）
```

> **注意**：
> - 声音学习（①）只需执行一次，后续可跳过。
> - LRC 文件**必须在执行 `lyrics-replace` 前准备好**，可通过 `lrc import` 或 `lrc extract` 获取。
> - LRC 文件一经准备即可长期复用，不需要每次重新提取。
> - 建议先完成歌词替换合成（④），再进行声音替换（⑤），以保证处理链路清晰、中间产物可复用。

## 3. 非功能需求

### 3.1 语言与国际化

| 阶段 | 支持语言 |
|---|---|
| 第一期（当前） | 中文 |
| 未来扩展 | 英文、日文等（架构需预留扩展点） |

系统设计时需将**语言处理模块**（语音识别、TTS、歌词对齐）设计为可替换/可插拔的组件，以便后续支持多语言时无需大规模重构。

### 3.2 性能

- 单首歌曲处理时长：**不超过歌曲本身时长的 3 倍**（如 3 分钟歌曲，处理时间不超过 9 分钟）
- 支持本地 GPU 加速（有 GPU 时自动启用）

### 3.3 易用性

- CLI 命令需有清晰的帮助文档（`--help`）
- 处理过程需有进度提示（如进度条或阶段日志）
- 错误信息需友好、可读，并给出可能的解决建议

### 3.4 版权与合规

- 暂不做版权限制处理（后续可根据需要补充）

## 4. 系统边界与假设

1. 本工具作为**本地命令行工具**运行，不涉及云端部署（第一期）
2. 输入音频假设为**已经混音完成的歌曲**（含人声和伴奏），工具需具备人声与伴奏分离的能力（作为内部处理步骤）
3. 系统假设输入音频的**人声部分为单一歌手**（暂不支持多人合唱的精确分离）
4. AI 生成歌词时，结构约束（段落数、句数、字数）可从用户提供的原歌词文本中提取
5. LRC 文件提取（路径二，ASR 自动提取）在歌唱场景下精度可能低于说话场景，提取后建议用户人工核对时间戳

## 6. 术语表

| 术语 | 说明 |
|---|---|
| LRC Extraction | LRC 提取，从歌曲音频中通过 ASR + 强制对齐算法自动识别歌词并生成带时间戳的 LRC 文件，或对已有 LRC 文件进行格式校验与规范化 |
| LRC File | LRC 歌词文件，包含每句歌词及其对应播放时间戳的文本文件（格式：`[MM:SS.xx]歌词`），是歌词替换合成的时间轴基准 |
| Voice Learning | 声音学习，通过提供参考音频让系统学习目标人声的音色特征，并以命名方式保存为可复用的声音模型 |
| Voice Replacement | 声音替换，利用已学习的声音模型，将歌曲中的人声音色进行替换，不改变伴奏与其他音频元素 |
| Voice Model | 声音模型，由声音注册过程生成的，描述特定人声音色特征的数据文件 |
| Lyrics Generation | 歌词生成，基于原歌曲的结构约束（段落数、句数、字数），由 AI 自动创作新歌词并输出为可编辑文本文件 |
| Lyrics Template | 歌词结构模板，仅包含段落和句数框架的空白歌词文件，供用户手动填写 |
| TTS | Text-to-Speech，文字转语音技术 |
| Vocal Separation | 人声分离，从混音歌曲中分离出纯人声和伴奏两个音轨 |
| Re-synthesis | 重合成，将新歌词按照原旋律时间轴重新生成演唱音频 |
| Lyrics Alignment | 歌词对齐，将歌词文本与音频时间轴进行匹配 |
| Forced Alignment | 强制对齐，在已知歌词文本的前提下，通过语音模型将每个字/词精确定位到对应的时间段 |
